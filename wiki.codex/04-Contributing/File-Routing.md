# Codex — File Routing

Use this page to decide which folder a new piece of content belongs in. The decision turns on three questions: *what is the content*, *who is it for*, and *what is its lifecycle*.

## Quick reference

| Content type | Folder | Lifecycle | Visibility default |
|---|---|---|---|
| Validated, curated page | `00-Start-Here/`, `01-Framework/`, `02-Domain/`, `03-Episodes/`, etc. | Long-lived; promoted through `gap → outlined → solid → ready` | `internal` |
| Working note, unvalidated idea | `_brain_dump/` | Quarantined; promoted to canon via explicit migration | `internal` |
| Newly-arrived source (not yet ingested) | `_inbox/` | Ephemeral; moved on ingest | `internal` |
| Ingested source (read-only archive) | `_sources/raw/` | Permanent; read-only | `internal` |
| Confidential page (project-internal, restricted distribution) | `_confidential/` | Long-lived; never shared externally | `confidential` |
| Decision log entry (append-only) | `_decisions/` | Permanent; append-only | `internal` |
| Auto-generated dashboard | `_dashboards/` | Derived; regenerated by `_scripts/update_dashboards.py` | `internal` |
| Wiki configuration | `_config/` | Long-lived; edited as policy changes | n/a (YAML, not pages) |
| Wiki-level operating context | `_context/` | Long-lived; mostly shipped verbatim from Codex | `internal` |
| Canonical reference YAML | `_canon/` | Long-lived; edited as canon is confirmed | n/a (YAML, not pages) |

## Decision tree

### 1. Is this a validated, curated page about a specific topic?

If yes → goes in a numbered content folder (`00-Start-Here/`, `01-Framework/`, `02-Domain/`, etc.). Pick the folder based on the page's `type` and subject:

- `type: overview` or orientation → `00-Start-Here/`
- `type: framework` (load-bearing structural rule) → `01-Framework/` (or whichever folder the project designates for framework pages)
- `type: reference`, `type: profile`, domain inventories → `02-Domain/` (or the project's reference folder)
- `type: episode` (sequential narrative or event log) → `03-Episodes/` (or the project's episode folder)
- `type: guide` (how-to-do-X) → wherever the audience-relevant folder lives, often `00-Start-Here/` or `04-Contributing/`
- `type: dashboard` → `_dashboards/` (auto-generated; do not author by hand)

If no → continue.

### 2. Is this a working note, unvalidated idea, or speculative brainstorm?

If yes → `_brain_dump/`. Use the `dump_status` frontmatter field (`exploring | validated | migrated | superseded | rejected`) to track maturity. Brain dump entries are explicitly NOT canon — validators flag any page that cites a brain dump as a `canon_source`.

Promote a brain dump to canon by creating a new page in the appropriate content folder, citing the brain dump in `dependencies` (not `canon_sources`), and updating the brain dump's `dump_status` to `migrated` with `migrated_to` pointing at the new page.

If no → continue.

### 3. Is this a raw external source?

If yes → routing depends on ingest state:

- **Not yet ingested** → `_inbox/`. Use reduced source-file frontmatter (spec §2.3). Files in `_inbox/` are ephemeral; the ingest procedure moves them to `_sources/raw/` on success.
- **Already ingested** → `_sources/raw/` (read-only). Do not author content here directly — the ingest flow places files. `canon_sources` on validated pages always cite `_sources/raw/` paths, never `_inbox/`.

See `_context/INGEST_PROCEDURE.md` for the ingest workflow.

If no → continue.

### 4. Is this content confidential — restricted to a small audience and never shared externally?

If yes → `_confidential/`. Set `visibility: confidential` in frontmatter. Validators enforce that confidential content never appears in public-facing exports.

If no → continue.

### 5. Is this a decision log entry (an immutable record of "we decided X on date Y")?

If yes → `_decisions/`. Append-only by convention; do not edit prior entries. `_decisions/ingest-log.md` is the canonical append-only log of ingest events.

If no → continue.

### 6. Is this configuration or canonical reference data (YAML, not a page)?

- **Validator policy** → `_config/` (e.g. `forbidden_terms.yaml`, `cascade_map.yaml`, `reveal_leak_patterns.yaml`, `steel_threads.yaml`, `concept_coverage.yaml`).
- **Canonical facts** (named entities, counts, taxonomies, timelines) → `_canon/` (e.g. `roster.yaml`, `counts.yaml`, `taxonomy.yaml`, `timeline.yaml`).

If no → continue.

### 7. Is this Claude-operating context (rules, procedures the agent needs at session start)?

If yes → `_context/`. Most files here ship verbatim from Codex and are overwritten by `_scripts/sync_from_kit.py`; the exception is `CLAUDE_CONTEXT_RULES.md`, which is project-owned.

If no → reconsider the content. If it doesn't fit any of the above, it may belong in a project-specific top-level folder; coordinate with the wiki maintainer before creating new top-level structure.

## Special-case clarifications

**Visibility ≠ folder.** A page can live in any content folder with any `visibility` value (`internal | public | confidential`). The folder determines lifecycle and curation discipline; `visibility` determines audience.

**`_brain_dump/` vs `_inbox/`.** Brain dumps are project-authored speculation; inbox entries are externally-sourced raw material. A note you wrote about a hypothesis goes in `_brain_dump/`; a PDF a collaborator sent you goes in `_inbox/`.

**`_decisions/` vs decision-record pages.** A `type: decision` page in a content folder is a long-form, public-facing decision write-up. `_decisions/*.md` is the append-only log for transient records (e.g. ingest events, configuration changes). Use the log for "what happened when"; use a `type: decision` page for "what we decided and why."
