# Delta Force — `_parse_yaml_subset` block-list fix

**Directive:** `dir-20260614qq-codex-parser-fix` (#2). The Codex YAML-subset FRONTMATTER parser
`_lib/frontmatter._parse_yaml_subset` silently drops block-style lists: a `canon_sources:` key
with empty inline value → `_parse_value("")` → `None`, and the following `  - "x"` lines have no
colon → `continue` (skipped). Block-list frontmatter values vanish. Latent because
`validate_canon_integrity` gates on `status: ready`. The sibling `parse_config_yaml` (for
`_config/*.yaml`) already handles block lists via a separate, richer state machine.

**DFTS:** Hard trigger — foundation module, Syncs to all consumers (breaking-contract risk).
Convened. Fable 5 unavailable → 5 seats + chairman on Opus 4.8.

**Decision framed:** (A) teach `_parse_yaml_subset` flat block-scalar-lists (look-ahead on
empty-value key) vs (B) inline-only lint that errors on block form.

## Council verdict

- **Breaker:** **Live data confirmed broken** — `wiki.codex/git/01-Architecture/Reference-Implementation.md`
  and `02-Operations/Orchestrator-Participation.md` (**status solid**) carry real block-style
  `canon_sources:` → `None` today. No page relies on the `None`; the fix is strictly more correct.
  Edge tests required: value-with-colon (safe — non-empty value, look-ahead never fires),
  item-with-colon (route via `_parse_value`, assert un-mangled), tab / 0-indent dashes (decide +
  test), comment between items (must not terminate), empty-key-then-scalar-key (don't over-consume),
  block-list-last-key at EOF (items not None). No existing test asserts `canon_sources == None`
  (grep clean). Reject lint — it leaves the two live pages broken + Sync re-ships block templates.
- **Simplifier:** **A, the ~15-line look-ahead. Not B, not reuse.** The current loop already
  ignores `- ` lines; the fix intercepts the empty-value branch. Forbidding the idiomatic YAML form
  to protect a weak parser is the engine dictating to users (and B needs the same detection logic,
  then errors instead of collecting — same code, worse outcome). Reusing the 150-line
  `parse_config_yaml` is over-coupling two parsers with deliberately different contracts.
- **Surgeon:** **Ship, tightly scoped — one file (docstring + `_parse_yaml_subset`) + its tests.**
  `parse_config_yaml` DO NOT TOUCH. Data migration across consumer wikis = OUT (separate follow-up;
  both forms parse identically after). New lint = nobody asked, cut it. Blast radius verified safe:
  all consumers use `fm.get(k) or []` / `if not fm.get(k)` (validate_canon_integrity:69,
  update_dashboards:276/338, collect_open_questions:43) — `None`/`[]` both falsy. Caveat: bare
  `key:` with no items must STILL return `None`; inline `[]` untouched.
- **Reviewer:** Approve, with: (1) the deeper smell is silent-`None` on unsupported shapes — argues
  unsupported frontmatter should RAISE; (2) pick a canonical form in spec §2.3 (inline default, block
  accepted) so there aren't two equal idioms; (3) update the docstring precisely — block SCALAR
  lists now in scope, nested mappings / list-of-mappings still OUT (don't overclaim).
- **Pragmatist:** **Parser-fix over lint, yes.** Pure-additive — the look-ahead trigger
  (empty-value key) is mutually exclusive with scalar/inline/None paths → near-zero regression.
  Tests first, then ~15-line change, then full suite (done = new tests green + only the 2 known
  TestSyncStamp CRLF fails). Do NOT convert real frontmatter (churn vs the revert-discipline; add a
  real-shape fixture instead). Config parser + lint OUT of scope.

## Chairman synthesis (Opus 4.8)

**Unanimous: parser-fix (A); reject the lint, config-parser reuse, data-migration, and any new
lint.** Implement a flat block-scalar-list look-ahead in `_parse_yaml_subset`:
- Trigger ONLY when a `key:` has empty inline value (mutually exclusive with every other path —
  this is the near-zero-regression guarantee).
- Collect consecutive indented (`indent > 0`) `- item` lines into a list via `_parse_value`;
  tolerate interleaved full-comment lines; terminate on blank or non-dash content; advance the
  line cursor past consumed lines (index-based loop).
- If no `- ` items follow → keep `None` (back-compat: bare `source:` stays `None`).
- Inline `[...]` and scalars never reach the look-ahead — untouched.

**Reviewer's two escalations — partially adopt, scope-guarded:**
- **RAISE on unsupported shapes — DEFER.** Turning silent-`None` into an error is a separate
  behavior change with its own blast radius (a page with a stray ignored line would suddenly raise).
  Out of scope for "fix the block-sequence bug"; logged as a follow-up. (Surgeon/Pragmatist/Simplifier
  minimalism wins.)
- **Canonical form — ADOPT, minimal.** Add a one-line note to spec §2.3: inline flow-list is the
  canonical/preferred frontmatter list form; block-style scalar lists are also parsed. Update the
  `frontmatter.py` module docstring precisely (block SCALAR lists in scope; nested mappings /
  list-of-mappings still out). No Style-Guide edit (avoid creep).
