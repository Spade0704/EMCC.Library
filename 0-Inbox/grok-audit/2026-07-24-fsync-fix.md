# Cert request — asset-registry fsync-fix (Windows EBADF; unblocks the FAILED CORE)

Fixes the exact defect YOUR Windows re-run correctly caught in
`2026-07-21-asset-registry-core` (FAILED `7e7e53c`, superseding false-PASS `3b87e43`):
`os.fsync` on a read-only reopened handle in `_move_asset` → `OSError [Errno 9] EBADF`
on Windows, ERRORing 20 tests + 3 fails in `tests.test_asset_registry`.

**Range** `5badaf0` (2 commits off base `7e7e53c`): fix `4763b0a` + Auditor verdict `5badaf0`.
Diff is confined to `_move_asset` (11 lines) + evidence + audit note.

**The fix**: copy through the writable temp handle we own — `os.fdopen(fd,"wb")` +
`shutil.copyfileobj(src, handle)` → `handle.flush()` → `os.fsync(handle.fileno())` →
`os.replace`. fsync now runs on a **writable** fd (the mkstemp fd, owned by fdopen). This
is byte-for-byte the durable-write pattern the module's own `atomic_write_text` (~line 175)
already uses. Crash-safe temp→fsync→replace ordering (§9.3) unchanged.
